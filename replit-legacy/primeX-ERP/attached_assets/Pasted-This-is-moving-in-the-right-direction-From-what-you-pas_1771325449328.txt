2) Ensure workflow versioning is actually used (not just tables)

On document create: create workflowInstance row and store workflowInstanceId on the doc

Engine loads transitions using (tenant_id, doc_type, workflow_version from instance)

✅ Action item:

Add workflow_instance_id column to every workflow-enabled document table.

3) Approval Matrix: finish the “multi-approval” behavior

You added approvalMode + requiredCount. Great — but you need one more concept:

“Approval progress”

If approvalMode = ALL_REQUIRED or requiredCount > 1, you must track who approved so far and whether the step is complete.

Two common patterns:

Pattern A (simpler): store approvals in workflow_history and compute count from it.
Pattern B (faster): add document_approvals table:

tenant_id, doc_type, doc_id, approval_level, approved_by, approved_at

✅ Action item:

Decide A or B now (A is fine for MVP; B is better at scale).

4) Locks: make them safe and deterministic

You added lockToken + force unlock. Make sure you enforce this rule:

Only the session holding the correct lockToken can save edits

“Force unlock” requires permission + reason and logs to audit trail

✅ Action item:

Don’t lock on “open view”. Lock only on “enter edit mode”.

Add lock_reason (optional) so users know why it’s locked.

5) Inventory: PARTIALLY_POSTED needs a “remaining qty” truth source

Adding PARTIALLY_POSTED is correct, but you must define what makes it partial.

✅ Action item:

For GRN/GIN lines, ensure there is:

posted_qty (or received_qty / issued_qty)

Status rules:

if sum(posted_qty) == 0 → not posted

if 0 < sum(posted_qty) < sum(ordered_qty) → PARTIALLY_POSTED

if sum(posted_qty) == sum(ordered_qty) → POSTED/CLOSED

6) SoD enforcement: define the exact rule you will enforce

You added actorRoleId and controlStep in history — good. Now pick the rule set:

Recommended default:

Maker cannot Check/Approve/Post the same document

Checker cannot Approve/Post

Same user cannot do two control steps on same doc (strict mode)

✅ Action item:

Put SoD decision into config: STRICT | STANDARD | OFF (per tenant, per doc_type).

Your approval queue dashboard should not just list documents — it must show:

why it’s in my queue

what action I can take now

what I’m missing (e.g., needs 2 approvals, has 1)

✅ Action item:

Implement the GET /documents/:type/:id/actions endpoint early and use it everywhere.
