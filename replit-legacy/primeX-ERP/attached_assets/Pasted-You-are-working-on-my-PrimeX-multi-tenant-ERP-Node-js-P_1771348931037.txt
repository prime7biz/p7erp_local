PART-1: MULTI-TENANT AUTH + APPROVAL + STRICT ISOLATION
========================================================

GOALS (Must):

3) Platform Super Admin approves/rejects tenants from a backend admin portal.
4) Only APPROVED tenants can log in and access the ERP.
5) Strict tenant data isolation: no tenant can ever read/write another tenant’s data.

TECH DECISIONS (Follow strictly):
- Use PostgreSQL schema-per-tenant isolation.
- Keep a PUBLIC schema (or “public”) for shared/platform tables (tenants, users, platform_admins, subscriptions, audit logs).
- Each tenant gets its own schema: tenant_<tenantIdOrCompanyCode>.
- All tenant business tables (orders, inventory, accounting, etc.) live inside that tenant schema.
- Application must set `search_path` per request based on authenticated tenant schema, and must never accept tenant_id or schema from client payload.

STACK:
- Node.js backend (Express recommended; if project already uses Fastify/Nest, adapt but keep same logic).
- Use `pg` (node-postgres) for DB access OR existing ORM if present.
  - If ORM cannot safely support schema-per-tenant with per-request search_path, use `pg` for tenant-scoped queries.
- JWT auth for tenant users and platform admins.
- bcrypt for password hashing.
- Use httpOnly cookies for refresh tokens if already implemented; otherwise simple JWT access token is ok for now.

Create these tables (or migrate existing to match):

- tenant_schema should be: "tenant_" + tenant.id (best) OR sanitized "tenant_" + company_code.

- Set email_verified=false initially; add email verification flow (below).

2) Email verification for tenant SUPER_ADMIN
Routes:
- POST /api/public/send-verification (email)

Behavior:
- Send verification link to super_admin_email using a signed token.
- Mark users.email_verified=true after successful verification.
- Tenant can still remain PENDING; approval is separate.

3) Platform Admin Login + Admin Portal (Protected)
- Seed one platform super admin from env vars:
  PLATFORM_ADMIN_EMAIL, PLATFORM_ADMIN_PASSWORD
Routes:
- POST /api/platform/auth/login  (returns platform admin JWT)
- GET  /api/platform/tenants?status=PENDING

- Optional: GET /api/platform/tenants/:id (details)

Approval behavior:
- Update tenants.status=APPROVED, approved_at, approved_by.
- Create tenant schema in Postgres: CREATE SCHEMA <tenant_schema>.
- Run tenant schema migrations/DDL for seed tables in that schema.
- Insert default settings rows in tenant.settings.
- Write audit_log entry.

Reject behavior:
- Update tenants.status=REJECTED, rejected_reason.
- Write audit_log entry.
- Do NOT create schema.

4) Tenant User Login
Route: POST /api/auth/login
Input: company_code, username, password
Behavior:
- Find tenant by company_code
- If tenant.status != APPROVED => deny with clear message ("Company is pending approval" / "Rejected" / "Suspended")
- Find user by tenant_id + username
- Verify password
- Issue JWT containing tenant_id, tenant_schema, user_id, role
- Return token + basic user profile

5) Password reset (at least for SUPER_ADMIN via email)
Routes:
- POST /api/public/forgot-password (email)
- POST /api/public/reset-password (token, new_password)
Behavior:
- Send reset token to email
- Reset password_hash on success

STRICT TENANT ISOLATION (Mandatory middleware + DB handling):
- For any tenant-protected route:
  - Verify JWT.
  - Derive tenant_schema ONLY from JWT (never from request body/query).
  - Use a per-request DB session:
    - Acquire client from pool
    - BEGIN
    - SET LOCAL search_path TO <tenant_schema>, public;
    - Execute all queries with this same client
    - COMMIT/ROLLBACK
    - Release client
- Ensure NO query can access another tenant schema.
- Block any direct schema usage coming from user inputs.
- Never accept tenant_id from client. Always use tenant_id from JWT.
